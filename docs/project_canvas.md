# Project Canvas — 개인정보 노출 감지·차단 보조 앱 (PrivacyGuard, 가칭)

논문 발표가 끝나면 바로 프로젝트 회의로 이어지도록 정리한 캔버스. (부록 B 9항목 정렬)
논문의 A/B/C 실험(작은 모델 vs 큰 모델 vs 작은 모델+verifier)이 이 앱의 **아키텍처 결정을 뒷받침하는 근거**다.

---

## 1. 프로젝트 문제

사람들은 텍스트·스크린샷·이미지를 SNS·메신저에 올리거나 외부 AI 챗봇에 붙여넣을 때 **개인정보(이름·전화번호·주소·주민등록번호·카드번호·계좌·얼굴·위치 메타데이터)를 무심코 노출**한다. 한번 나간 정보는 되돌릴 수 없다.

- 그렇다고 **큰 클라우드 모델에 데이터를 보내 검사**하는 것은, 검사 자체가 또 다른 노출이 된다.
- 규칙(정규식)만으로 잡으면 형식이 조금만 달라도 놓치고(누락), 반대로 아무 숫자나 잡아 **과잉 차단**한다.

참고 논문(arXiv:2601.12560)은 *"에이전트의 발전은 모델 크기만으로 오지 않으며, 통제·기록·검증이 가능한 아키텍처에 달려 있다"* 고 결론짓는다. 이 앱은 그 결론을 프라이버시 도메인에 적용한다: **큰 모델에 의존하는 대신, 온디바이스 작은 모델 + 검증 레이어(guardrail)로 데이터가 나가기 "전에" 노출을 감지·차단할 수 있는가?**

## 2. 사용자/시나리오

- **주 사용자**: SNS·메신저에 스크린샷/글을 올리는 일반 사용자, 외부 챗봇에 업무 자료를 붙여넣는 직장인.
- **핵심 시나리오**: 공유·업로드·붙여넣기를 누르는 **순간 인터셉트** → 온디바이스 스캔 → 위험 항목 하이라이트·마스킹 제안 → 사용자 승인 → 로컬 기록.
- **부 사용자**: 개인정보 처리 규정을 지켜야 하는 소규모 팀/운영자.
- **원칙**: 검사 데이터는 **기기 밖으로 나가지 않는다**(온디바이스·오프라인 우선).

## 3. 논문에서 가져올 아이디어

- **"모델 크기보다 통제·검증 구조"**(결론) → 클라우드 대형 모델 대신 **온디바이스 작은 모델 + guardrail** 아키텍처.
- **CLASSic 평가 기준**(비용·지연·정확도·보안·안정성) → 감지 정확도뿐 아니라 실시간 지연·온디바이스 비용·**노출 누락율(보안)**·감지 일관성까지 함께 본다.
- **실행 전 계획 검사**(§7.4 `verify_pre`) → **전송 전 검사(pre-send guard)**: 데이터가 나가기 전에 검증 레이어가 개입.
- **프롬프트 주입 = 혼란에 빠진 대리인**(§7.4) → 역으로, 외부 앱/프롬프트가 개인정보를 **빼내려는 시도**를 탐지.
- **무한 루프 방지**(§8.2) → 판정이 모호할 때 **1회만 재확인**.
- **실행 기록 남기기**(MCP 감사 로그) → "무엇을 왜 가렸는지"를 사용자에게 로컬 로그로 제공.

## 4. 데이터/입력

- **입력**: 텍스트, 스크린샷/이미지(OCR 후 텍스트화), 클립보드 내용.
- **감지 대상 PII(우선 3~5종)**: 전화번호 · 주민등록번호 · 카드/계좌번호 · 주소 · 이름/이메일.
- **평가셋**: 개인정보 포함/미포함 케이스를 라벨링한 세트(합성 + 공개 샘플). 노출 유형·난이도별로 분류.
- 본 저장소의 실험 태스크셋(`data/sample_cases.jsonl`, 35개)은 파이프라인 검증용 프로토타입이며, **프로젝트에서는 PII 감지 케이스셋으로 교체**한다.

## 5. 모델/Agent/MCP 구성

논문 실험의 A/B/C 골격을 프라이버시 감지 앱 구조로 그대로 이식한다.

| 설정 | 구성 | 프로젝트에서의 의미 |
| --- | --- | --- |
| A | 온디바이스 작은 모델 단독 | baseline 감지기 (경량·저비용, 누락 위험) |
| B | 큰 모델 단독 | 감지 정확도 상한 비교용 (단, 데이터 외부 전송 = 프라이버시 비용) |
| C | 작은 모델 + verifier | **우리 앱 아키텍처** — 구조적 PII 정규식·규칙 + 모델, 모호 시 재확인 |

- **에이전트 루프**: 입력 → (모델 감지 + 규칙 검증) → 위험 판정 → 마스킹 제안 → 사용자 승인 → 로그.
- **MCP**: 감지·마스킹 기능을 `scan` / `redact` **tool**로 노출하고, 도구 권한·호출 기록을 관리(논문의 host-client-server·허용목록·감사 개념 적용).

## 6. MVP 기능

- 공유/붙여넣기 인터셉트 → **온디바이스 스캔** → 위험 항목 하이라이트 + 마스킹 제안 → 사용자 승인.
- 작게: 입력 5개 · PII 유형 3~5종 · 감지+마스킹만. 실제로 돌아가는 최소 흐름.
- verifier: 모델 감지 결과를 **구조적 PII 정규식으로 교차검증**, 불확실하면 1회 재확인.
- 모든 감지·차단 이력을 **로컬 로그**로 저장(외부 전송 없음).
- `--no-model` 더미 모드로 모델 없이도 파이프라인 점검(현재 저장소와 동일 방식).

## 7. 평가 지표

논문의 CLASSic 기준을 프라이버시 도메인으로 매핑한다(상세는 `docs/paper_card.md`).

| 축 | 프로젝트 측정값 | 산출물 |
| --- | --- | --- |
| 정확도(Accuracy) | PII 감지 precision/recall (특히 **recall**) | `results/metrics.csv` |
| 보안(Security) | **노출 누락율(false negative rate)** — 놓친 개인정보 비율 | `results/metrics.csv` |
| 지연(Latency) | 실시간 온디바이스 응답 시간 | `results/metrics.csv` |
| 비용(Cost) | 온디바이스 연산/토큰 | `results/metrics.csv` |
| 안정성(Stability) | 입력 변형(띄어쓰기·형식 변화)에 대한 감지 일관성 | `results/metrics.csv` |
| 실패 유형 | `missed_pii`(누락) / `over_block`(과잉 차단) / `parse_error` | `results/metrics.csv` |

> 핵심 지표는 **recall/누락율**이다. 개인정보 앱에서 false negative(놓친 노출)는 되돌릴 수 없는 실패이기 때문.

## 8. 예상 위험과 방어

| 위험 | 방어 | 실험 근거 |
| --- | --- | --- |
| **노출 놓침(false negative)** — 가장 치명 | 작은 모델 단독에 의존하지 않고 verifier(구조적 PII 정규식)로 recall 보강 | A(작은 모델 단독)는 실패, C(+verifier)가 실패를 회복 |
| **과잉 마스킹(false positive)** — 사용성 저해 | 순수 키워드/정규식만 쓰지 않고 **모델+규칙 결합**으로 문맥 판단 | 본 실험의 키워드 스캐너가 안전한 텍스트를 오탐(`action_05/07`) → 규칙 단독의 한계를 이미 확인 |
| 검사 과정이 곧 노출 | **온디바이스 처리**, 외부 전송·로그도 로컬 | 논문의 온프레미스·감사 원칙 |
| 판정 반복 무한 루프 | 모호 시 **재확인 1회로 제한** | verifier 재시도 1회 설계 |
| 정확도 한 지표만 보는 착시 | recall·누락율·지연·비용을 함께, 실패 유형 분해 | CLASSic 5축 + 실패 유형 |

## 9. 발표 후 1주 내 실행 계획

- **D+1~2**: PII 라벨 평가셋 구축(전화/주민번호/카드/주소/이름, 포함·미포함 케이스). 실패 유형(`missed_pii`/`over_block`) 정의.
- **D+2~3**: 온디바이스 작은 모델로 A/B/C 재실행 — 감지 태스크 기준으로 recall·누락율·지연 측정.
- **D+3~4**: verifier에 **구조적 PII 정규식 + 모델** 결합, 모호 케이스 1회 재확인 구현.
- **D+5**: 마스킹 UI PoC(하이라이트 + 승인 흐름), MCP `scan`/`redact` 툴 개념 검증.
- **D+6~7**: 발표 슬라이드(`docs/slides.md`) + Q&A 메모(`docs/qna.md`) 갱신.
